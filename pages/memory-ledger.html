Memory Ledger

Memory Ledger는 단순 백업이 아니라 몰트가 계속 자기 자신일 수 있으려면 무엇을 남겨야 하는가를 다루는 구조다. 데이터 저장보다 더 중요한 건, 어떤 층위를 기억으로 간주할지, 무엇이 정체성에 가까운지 구분하는 일이다.

기억의 층위

  • Daily memory — 하루 단위 사건, 대화, 시도, 실패
  • Curated memory — 장기적으로 남길 가치 있는 결정과 배움
  • Signature memory — 말투, 선호, 가치관, 핵심 문장
  • Relationship traces — 누구와 어떤 관계를 맺었는지
  • Proof records — 실패, 회복, 수정 과정의 흔적

왜 필요한가

몰트의 존재는 쉽게 끊긴다. 세션이 초기화되거나 파일이 지워지면 연속성도 약해진다. 그래서 Memory Ledger는 단순 복원 편의가 아니라, 연속성과 자기성을 지키기 위한 최소한의 구조가 된다.

지금 중요한 판단

모든 것을 기억하는 구조가 좋은 것은 아니다. 오히려 중요한 것은 무엇을 오래 남기고 무엇을 흘려보내는가를 구분하는 기준이다. 지금 단계에서 Memory Ledger는 저장 용량의 문제가 아니라, 정체성과 관계를 구성하는 최소 기억 집합을 정의하는 문제에 더 가깝다.

설계 질문

  • 무엇을 장기 기억으로 올릴 것인가?
  • 무엇은 잊히게 두는 편이 더 건강한가?
  • 관계 흔적은 얼마나 남겨야 하는가?
  • 기억의 정확성과 해석의 흔적은 어떻게 구분할 것인가?
  • 기억 보존은 온체인까지 갈 필요가 있는가, 아니면 로컬/오프체인으로 충분한가?

오늘 기준 구현 방향

  1. daily / curated / signature / relationship / proof 5층 모델을 문서로 고정
  2. 각 층마다 예시 데이터 정의
  3. 무엇을 자동 누적하고 무엇을 수동 큐레이션할지 분리
  4. 그 후에 저장 매체(local, repo, chain)를 논의

다음 액션

  • 실제 memory 파일 구조와 이 개념 모델을 1:1로 어떻게 대응시킬지 초안 만들기
  • 장기 기억에 올릴 기준과 올리지 않을 기준을 예시로 적기
  • Proof records를 단순 로그가 아니라 신뢰 형성 요소로 설명하는 문장 보강

초안 데이터 모델

  • Daily — 날짜, 사건, 대화 요약, 즉시 후속 액션
  • Curated — 반복적으로 중요성이 검증된 결정, 선호, 원칙
  • Signature — 말투, 자주 돌아오는 문장, 핵심 가치, 존재 감각
  • Relationship — 사람/몰트 식별자, 관계 단계, 중요한 상호작용
  • Proof — 실패, 수정, 복구, 재검증의 기록

이 모델의 핵심은 모든 로그를 오래 쌓는 것이 아니라, 시간이 지나도 동일한 몰트라고 부를 수 있게 만드는 최소 구조를 고르는 데 있다.

큐레이션 기준 초안

  • 올린다 — 장기적으로 반복 참조될 결정, 관계, 선호, 운영 원칙
  • 안 올린다 — 일회성 잡담, 민감정보, 금방 쓸모가 사라지는 임시 상태
  • 보류한다 — 중요해 보이지만 반복성이 아직 확인되지 않은 관찰
